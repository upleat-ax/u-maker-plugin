01. U-MAKER란?

한줄 요약: 10개의 AI 에이전트가 PBGD 4-Phase(Plan → Build → Gatekeeping → Deploy)를 따라, 기획 자료에서 설계 → 코드 → 테스트 → 배포까지 자동으로 수행하는 Claude Code 플러그인입니다.

기존 방식 vs U-MAKER

기존 방식U-MAKER 방식
문서코드 먼저, 문서는 나중에문서 먼저, 코드는 문서 기반 생성 Docs-First
기존 프로젝트문서 없이 코드만 존재/u-reverse로 코드에서 SSoT 역공학
데이터 분석200페이지 RFP를 한 번에 → loss3-Layer chunk 분석 → 손실 없음
요구사항 추적스프레드시트 수동 관리4-Tier ID(USR→FR→US→FT) 자동 추적
설계 변경SRS 수정 시 수동 갱신Auto-Cascade 자동 전파
품질개발 후 수동 QAPhase Gate 자동 검증 (11개 기준, threshold 95)
AI 판단블랙박스Assumptions Log로 기록 + 리뷰

핵심 개념 5가지

1 SSoT (Single Source of Truth)

모든 결정이 .u-maker/ 아래 문서에 기록됩니다. .md(사람용) + .json(기계용) 2종이 항상 함께 생성되어 일관성을 보장합니다.

2 3-Layer 파이프라인

data/dropzone/(Raw) → data/digest/(Structured JSON) → docs/(Deliverables)
data/dropzone/에 파일을 넣으면 Plan 실행 시 자동 분석합니다. 200페이지 RFP도 chunk 단위로 분석해 손실 없이 처리합니다.

3 PBGD 4-Phase

Plan → Build(Design↔Dev) → Gatekeeping(Doc+QA) → Deploy. 각 전환 시 Gatekeeper가 --loop [N]으로 11개 기준을 검증합니다 (pass ≥95, deploy-gate ≥98).

4 4-Tier ID 체계

USR(사용자) → FR(기능 요구사항) → US(유저 스토리) → FT(Feature = 구현 단위). 원시 요구사항부터 코드까지 전 구간 자동 추적합니다.

5 Assumptions Log

AI가 정보 부족 시 추정한 내용을 .u-maker/.state/assumptions.json에 자동 기록합니다. -i 모드에서 리뷰 후 승인/거부할 수 있으며, 거부 시 관련 문서에 cascade re-evaluation이 트리거됩니다.

PBGD 4-Phase (Plan, Build, Gatekeeping, Deploy): --loop [N]으로 N개 기준(기본 5, 최대 11) 채점. pass ≥95 / deploy-gate ≥98을 통과해야 다음 Phase로 진행

AI 팀 구성 (10 Agents)

10-Agent Architecture: PM Agent(u-agent-pm)가 전체를 조율하며, 9개 전문 에이전트(plan, build, design, dev, qa, gatekeeper, deploy, figma, report)에 작업을 분배

03. 프로젝트 시작하기

U-MAKER는 세 가지 방식으로 프로젝트를 시작할 수 있습니다.

3.1 Forward Engineering (새 프로젝트)

기획 자료부터 시작해서 설계 → 코드 → 테스트까지 단계별로 진행합니다.

Step 1A — 새 프로젝트 시작

/u-createproject my-saas — Turborepo+Bun 모노레포 스캐폴딩 + /u-prepare 자동 실행

⚠️ 생성된 프로젝트 폴더(my-saas/)로 이동 후 Claude CLI를 다시 시작해야 합니다: cd my-saas && claude
Step 1B — 기존 프로젝트에 적용

/u-prepare my-saas (별칭: /u-init) — foldertree + dropzone + analyze/reverse + 요구사항 협의까지 포함하는 umbrella

Step 2 — 자료 수집 (Plan Phase)

.u-maker/data/dropzone/에 RFP, 회의록, AS-IS 자료, Figma URL을 드롭. /u-prepare가 자동으로 /u-analyze (또는 Figma 소스 감지 시 /u-tools-figma) 를 위임합니다.

Step 3 — 기획 (Plan Phase)

/u-plan --app myapp — digest → SRS + IA 생성 (+ 선택적 wireframe)

Step 4 — 와이어프레임 (선택)

/u-wireframe --app myapp --all — screens.json 기반 HTML 와이어프레임 (6-Section 템플릿)

Step 5 — 빌드 (Build Phase)

/u-build --app myapp — Design(ERD/API/Screens/DS) ↔ Dev(FE/BE/DB) ping-pong 오케스트레이션. --only fe|be|db 또는 /u-design --only erd로 세분화 가능

Step 6 — 검증 (Gatekeeping Phase)

/u-gatekeeping --app myapp (별칭 /u-check, /u-qa) — 문서 scoring(11 criteria, pass ≥95) + 런타임 QA(TC + Test Results)

Step 7 — HTML 산출물

/u-output --app myapp — SSoT 문서를 독립 실행형 HTML로 변환

Step 8 — 배포 (Deploy Phase)

/u-deploy --app myapp — 대화형 타겟(GitHub Actions / Vercel / Docker) + 아티팩트 선택, deploy-gate ≥98, continuous regeneration

자동 실행: /u-loop --app myapp를 사용하면 Plan → Build → Gatekeeping → Deploy 전체를 gatekeeper 검증과 함께 무인으로 자동 실행합니다.
선행조건 자동 캐스케이드: 선행 Phase 문서가 없으면 자동으로 호출됩니다.
/u-build 실행 시 Plan 문서 없으면 → /u-plan 자동 호출. /u-dev는 Design 문서 없으면 → /u-design 자동 호출.
즉, /u-build 하나로 Plan → Build 전체가 자동 실행될 수 있습니다.

3.2 Reverse Engineering (기존 프로젝트)

이미 코드가 있지만 SSoT 문서가 없는 프로젝트를 위한 워크플로우입니다.

# 1. 프로젝트 초기화 (/u-init 은 /u-prepare 별칭)
/u-prepare my-existing-app

# 2. 소스 코드 → digest 역공학
/u-reverse --app my-app              # Code → digest → Plan → Build 전체 역생성
/u-reverse --app my-app --db prisma/  # DB 스키마 경로 수동 지정
/u-reverse --app my-app --loop       # Gatekeeper 검증 포함 (기본 N=5)

# 3. Plan → Build 파이프라인 진행
/u-plan --app my-app
/u-build --app my-app

# 4. v3.x PDCA 프로젝트 마이그레이션 (docs/{app}/check/ → gatekeeping/)
/u-prepare-foldertree --app my-app --migrate

3.3 자연어로 시작하기

명령어를 몰라도 자연어로 말하면 PM Agent가 자동으로 적절한 커맨드로 라우팅합니다.

# 자연어 예시
"쇼핑몰 프로젝트 시작해줘"          # → /u-prepare + /u-plan 자동 라우팅
"ERD 만들어줘"                     # → /u-build --only erd (또는 /u-design --only erd)
"테스트 돌려줘"                    # → /u-gatekeeping (또는 /u-qa)
"이 코드 분석해서 문서 만들어줘"     # → /u-reverse
"Figma 파일 분석해줘"              # → /u-tools-figma
"배포 준비해줘"                    # → /u-deploy
Tip: 자연어 라우팅이 작동하지 않으면 슬래시 커맨드를 직접 사용하세요. PM Agent는 26개 스킬(별칭 3개 포함)을 모두 인지합니다.

04. Interaction Mode

모든 Phase 커맨드에 아래 세 가지 상호작용 모드를 적용할 수 있습니다.

Auto default

/u-plan [app] — 플래그 없으면 auto. 빠르게 진행하며, 판단이 필요한 사항은 Assumptions Log에 기록합니다.
적합: 빠른 프로토타이핑, 야간 무인 실행, 반복 작업

Interactive -i

/u-plan [app] -i — 분기점에서 사용자에게 질문합니다.
적합: 중요 설계 결정이 필요할 때, 첫 번째 프로젝트 setup

Step --step

/u-plan [app] --step — 매 단계마다 확인을 요청합니다.
적합: 학습 목적, 산출물 품질 세부 검토, 디버깅

참고: auto 모드에서 maxAssumptions 설정값을 초과하면 자동으로 interactive로 전환됩니다. Assumptions Log(.u-maker/.state/assumptions.json)를 리뷰하고 -i로 재실행하면 auto로 복귀합니다.

alias		 alias → /u-prepare (backward-compat with v3.x). 신규 프로젝트는 /u-createproject 또는 /u-prepare 사용 권장 /u-init my-saas

07. /u-discuss 협업 세션

다자간 토론 세션으로 구조화된 협업을 수행하고, 결과를 SSoT에 자동 반영합니다.

타입용도예시
brainstorm아이디어 발산. 자유로운 제안 수집/u-discuss brainstorm "결제 UX"
review산출물 검토. SSoT 문서 품질 리뷰/u-discuss review
decision기술/비즈니스 의사결정. 선택지 비교 + 결정/u-discuss decision "DB 선택"
workshop다단계 작업. 복잡한 주제를 단계별로 깊이 탐구/u-discuss workshop "메인 화면"
retroIteration 회고. 잘된 점/개선점/액션 아이템/u-discuss retro
Micro-commands (세션 중 사용):
  • @planner / @builder / @gatekeeper / @all — 특정 에이전트에게 질문
  • /idea [text] — 아이디어 태깅
  • /decide [text] — 결정사항 기록
  • /concern [text] — 우려/리스크 기록
  • /action [who] [text] — 액션 아이템 배정
  • /wrap — 세션 종료 + 결과물 SSoT 자동 적재
세션 결과 반영: /wrap으로 종료하면 아이디어는 백로그에, 결정사항은 관련 문서에, 액션 아이템은 태스크로 자동 등록됩니다.

08. 실전 시나리오

Scenario 1: 새 프로젝트 처음부터 끝까지 (PBGD)

/u-createproject my-saas → cd my-saas && claude → dropzone에 자료 드롭 → /u-plan --app myapp → /u-build --app myapp → /u-gatekeeping --app myapp → /u-deploy --app myapp

/u-createproject는 /u-prepare를 자동 포함합니다. 생성 후 새 폴더에서 Claude CLI를 재시작하세요.
한 줄로: /u-loop --app myapp — 전체 PBGD 파이프라인 자동 실행

Scenario 2: 기존 프로젝트에 SSoT 도입

/u-prepare my-app (별칭 /u-init) → /u-reverse --app myapp → /u-plan --app myapp → /u-build --app myapp → /u-gatekeeping --app myapp

Scenario 3: Figma 소스 기반 설계

Figma URL을 .u-maker/data/dropzone/에 드롭 → /u-prepare --app myapp (자동으로 /u-tools-figma 위임) → /u-plan → /u-build

v4.0.0-alpha.14은 reduced extraction path (source.pipeline == "reduced"). downstream은 coverageWarnings[]를 warning으로 처리합니다.

Scenario 4: PR 생성

/u-tools-git-pr — 현재 브랜치의 변경사항을 분석하여 커밋 그루핑 + 구조화된 PR body + 리뷰 가이드를 자동 생성합니다.

Scenario 5: Phase Gate 미통과 대응

/u-gatekeeping --loop 11 → FAIL (스코어카드 출력) → 미충족 항목 확인 → 해당 문서 수정 → 재실행

예: IA 미완 → /u-plan --app myapp --only ia → /u-gatekeeping --app myapp --loop

Scenario 6: 야간 무인 실행

/u-loop --app myapp — 밤새 돌려놓으면 아침에 결과물을 확인할 수 있습니다.
--from plan --to gatekeeping으로 특정 구간만 지정 가능. 체크포인트 기반 재개 및 에러 자동 복구 지원.

Scenario 7: v3.x PDCA → v4.x PBGD 마이그레이션

/u-prepare-foldertree --app my-app --migrate — docs/{app}/check/를 gatekeeping/으로 리네임. 기존 SSoT는 그대로 유지되며, 이후 /u-build, /u-gatekeeping, /u-deploy를 정상 사용 가능.

10. Quality Gate (11 Criteria)

Gatekeeper 에이전트가 11개 범용 criteria로 문서/코드를 채점합니다. --loop [N]으로 N개 선택(기본 5, 최대 11). pass threshold 95, deploy-gate threshold 98. 미달 시 개선 항목이 심각도 순으로 표시됩니다.

#Criterion검증 내용우선순위
1Completeness필수 섹션/항목이 빠짐없이 존재하는지 확인필수 (default N=5)
2Accuracy내용이 원본 digest 및 상위 문서와 일치하는지 확인필수
3ConsistencyID, 용어, 수치가 문서 간 일관되는지 확인필수
4TraceabilityFR → US → FT → TC 체인이 빈틈없이 완성되는지 확인필수
5TOC Quality목차 논리성, 적절한 깊이, 누락 확인필수
6Content Composition논리적 구조, 자연스러운 흐름, 핵심 정보 배치확장 (6–11)
7Visual Adequacy테이블, 리스트, 코드 블록 등 시각 요소 적정성확장
8Diagram Fitness다이어그램 유형 적합성, SVG 렌더링 확인확장
9Mermaid IntegrityMermaid 문법 오류 없음, 노드/엣지 완전성확장
10JSON Sync.md와 .json의 항목 수, ID, 상태 동기화 확인확장
11Cross-Referencelinks.json과 실제 문서 참조의 일치성 확인확장
Gate 명령어: 각 Phase 명령에 --loop [N]을 붙여 실행하면 스코어카드가 출력되고 PASS(≥95점, deploy는 ≥98점) 시 다음 Phase로 전환됩니다. FAIL 시 미충족 항목이 심각도 순으로 표시되며, 최대 3회 자동 개선 반복합니다.

11. FAQ

  • Q
    명령어가 인식되지 않아요
    Claude Code를 재시작해보세요. 그래도 안 되면 claude plugin add upleat-ax/u-maker-plugin으로 재설치하세요.
  • Q
    어떤 명령어를 써야 할지 모르겠어요
    자연어로 말하세요. PM Agent(u-agent-pm)가 26개 스킬(별칭 3개 포함) 중 가장 적합한 명령으로 자동 라우팅합니다.
  • Q
    기존 프로젝트에 적용하려면?
    /u-prepare my-app (별칭 /u-init) → /u-reverse --app my-app로 코드에서 digest를 역공학하세요. --loop 옵션으로 Gate 검증 포함 가능.
  • Q
    Figma 파일로 시작하고 싶어요
    Figma URL을 dropzone에 넣고 /u-prepare를 실행하세요. /u-tools-figma가 자동 위임됩니다. 또는 직접 /u-tools-figma <url>도 가능합니다. v4.0.0-alpha.14은 reduced extraction path를 사용합니다.
  • Q
    문서 언어를 바꾸고 싶어요
    u-maker.config.json의 language.documents를 "en", "ja", "zh" 등으로 변경하세요.
  • Q
    Gate가 계속 FAIL이에요
    스코어카드의 미충족 항목을 확인하세요. pass threshold 95, deploy-gate 98입니다. 해당 문서를 수정하거나 --only로 특정 산출물만 재생성한 후 --loop로 재검증하세요.
  • Q
    /u-loop 실행 중 에러가 나면?
    체크포인트 기반 재개를 지원합니다. 동일한 /u-loop 명령을 다시 실행하면 .state/loop-state.json의 마지막 체크포인트부터 이어서 진행합니다.
  • Q
    PR을 빠르게 만들고 싶어요
    /u-tools-git-pr — 현재 브랜치의 변경사항을 분석하여 커밋 그루핑 + 구조화된 PR을 자동 생성합니다.
  • Q
    v3.x PDCA에서 v4.x PBGD로 마이그레이션하려면?
    /u-prepare-foldertree --app my-app --migrate를 실행하세요. docs/{app}/check/를 gatekeeping/으로 리네임하고 config를 v4.0으로 업데이트합니다. 기존 SSoT는 그대로 유지됩니다.
  • Q
    /u-deploy가 아티팩트를 재생성하지 않아요
    Manifest에 기록된 sourceHashes와 현재 SSoT가 동일하면 재생성을 건너뜁니다. SSoT 문서를 수정하면 on-deploy-state.js hook이 .state/deploy-stale.json에 drift를 기록하여 다음 /u-deploy가 재생성합니다.
