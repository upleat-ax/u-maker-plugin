01. U-MAKER란?

한줄 요약: 7개의 AI 에이전트가 PDCA 5-Phase(Plan → Design → Dev → Check → Ship)를 따라, 기획 자료에서 설계 → 코드 → 테스트까지 자동으로 수행하는 Claude Code 플러그인입니다.

기존 방식 vs U-MAKER

기존 방식U-MAKER 방식
문서코드 먼저, 문서는 나중에문서 먼저, 코드는 문서 기반 생성 Docs-First
기존 프로젝트문서 없이 코드만 존재/u-reverse로 코드에서 SSoT 역공학
데이터 분석200페이지 RFP를 한 번에 → loss3-Layer chunk 분석 → 손실 없음
요구사항 추적스프레드시트 수동 관리4-Tier ID(USR→FR→US→FT) 자동 추적
설계 변경SRS 수정 시 수동 갱신Auto-Cascade 자동 전파
품질개발 후 수동 QAPhase Gate 자동 검증 (11개 기준, threshold 95)
AI 판단블랙박스Assumptions Log로 기록 + 리뷰

핵심 개념 5가지

1 SSoT (Single Source of Truth)

모든 결정이 .u-maker/ 아래 문서에 기록됩니다. .md(사람용) + .json(기계용) 2종이 항상 함께 생성되어 일관성을 보장합니다.

2 3-Layer 파이프라인

data/dropzone/(Raw) → data/digest/(Structured JSON) → docs/(Deliverables)
data/dropzone/에 파일을 넣으면 Plan 실행 시 자동 분석합니다. 200페이지 RFP도 chunk 단위로 분석해 손실 없이 처리합니다.

3 PDCA 5-Phase

Plan → Design → Dev → Check → Ship → 다음 Iteration. 각 전환 시 Gatekeeper가 11개 기준으로 Gate 검증을 수행합니다.

4 4-Tier ID 체계

USR(사용자) → FR(기능 요구사항) → US(유저 스토리) → FT(Feature = 구현 단위). 원시 요구사항부터 코드까지 전 구간 자동 추적합니다.

5 Assumptions Log

AI가 정보 부족 시 추정한 내용을 기록합니다. /u-assume approve/reject로 리뷰하며, reject 시 관련 문서에 cascade re-evaluation이 트리거됩니다.

PDCA 5-Phase (Plan, Design, Dev, Check, Ship): 11개 Gate Criteria (threshold 95)를 통과해야 다음 Phase로 진행

AI 팀 구성 (7 Agents)

7-Agent Architecture: PM Agent(u-agent-pm)가 전체를 조율하며, 6개 전문 에이전트(plan, design, dev, qa, gatekeeper, report)에 작업을 분배

03. 프로젝트 시작하기

U-MAKER는 세 가지 방식으로 프로젝트를 시작할 수 있습니다.

3.1 Forward Engineering (새 프로젝트)

기획 자료부터 시작해서 설계 → 코드 → 테스트까지 단계별로 진행합니다.

Step 1 — 초기화

/u-init my-saas — .u-maker/ 구조 생성, u-maker.config.json 초기화

Step 2 — 자료 수집

.u-maker/data/dropzone/에 RFP, 회의록, AS-IS 자료 드롭

Step 3 — 기획 (Plan Phase)

/u-plan --app myapp — dropzone 자료 분석(ingest 포함) → SRS + IA + Roadmap 생성

Step 4 — 설계 (Design Phase)

/u-design --app myapp — ERD + API Contract + Screens + ScreenFlow + Design System + RTM

Step 5 — 와이어프레임

/u-wireframe --app myapp --all — screens.json 기반 HTML 와이어프레임 (6-Section 템플릿)

Step 6 — 구현 (Dev Phase)

/u-dev --app myapp — FE + BE + DB 코드 생성 (--only fe|be|db 개별 지정 가능)

Step 7 — 검증 (Check Phase)

/u-check --app myapp — TC 설계 + 테스트 실행 + 결함 분류 + 리포트

Step 8 — HTML 산출물

/u-output --app myapp — SSoT 문서를 독립 실행형 HTML로 변환

Step 9 — 배포 (Ship Phase)

/u-ship --app myapp — 최종 검증 + iteration log + retrospective

자동 실행: /u-loop --app myapp를 사용하면 Plan → Design → Dev → Check → Ship 전체를 gatekeeper 검증과 함께 무인으로 자동 실행합니다.
선행조건 자동 캐스케이드: 선행 Phase 문서가 없으면 자동으로 호출됩니다.
/u-dev 실행 시 Design 문서 없으면 → /u-design 자동 호출 → Plan 문서 없으면 → /u-plan 자동 호출.
즉, /u-dev 하나로 Plan → Design → Dev 전체가 자동 실행될 수 있습니다.

3.2 Reverse Engineering (기존 프로젝트)

이미 코드가 있지만 SSoT 문서가 없는 프로젝트를 위한 워크플로우입니다.

# 1. 프로젝트 초기화
/u-init my-existing-app

# 2. 소스 코드 → SSoT 역공학
/u-reverse --app my-app              # Code → Design (ERD, API, Screens, DS) → Plan (SRS, IA) 전체 역생성
/u-reverse --app my-app --db prisma/  # DB 스키마 경로 수동 지정
/u-reverse --app my-app --loop [N]  # Gatekeeper 검증 포함 (N 기본값 5)

# 3. 역공학 결과 검증 + 보완
/u-sync [app]                       # 문서 간 정합성 검증 (13개 교차 검증 규칙)
/u-doc [app] srs --edit             # SRS에 비즈니스 의도 보완

3.3 자연어로 시작하기

명령어를 몰라도 자연어로 말하면 PM Agent가 자동으로 적절한 커맨드로 라우팅합니다.

# 자연어 예시
"쇼핑몰 프로젝트 시작해줘"          # → /u-init + /u-plan 자동 라우팅
"ERD 만들어줘"                     # → /u-design --only erd
"테스트 돌려줘"                    # → /u-check
"이 코드 분석해서 문서 만들어줘"     # → /u-reverse
"지금 프로젝트 상태가 어때?"        # → /u-status
"궁금한 게 있어"                   # → /u-ask
Tip: 자연어 라우팅이 작동하지 않으면, /u-ask로 U-MAKER에 직접 질문하세요. 프로젝트 맥락을 이해한 답변을 제공합니다.

04. Interaction Mode

모든 Phase 커맨드에 아래 세 가지 상호작용 모드를 적용할 수 있습니다.

Auto default

/u-plan [app] — 플래그 없으면 auto. 빠르게 진행하며, 판단이 필요한 사항은 Assumptions Log에 기록합니다.
적합: 빠른 프로토타이핑, 야간 무인 실행, 반복 작업

Interactive -i

/u-plan [app] -i — 분기점에서 사용자에게 질문합니다.
적합: 중요 설계 결정이 필요할 때, 첫 번째 프로젝트 setup

Step --step

/u-plan [app] --step — 매 단계마다 확인을 요청합니다.
적합: 학습 목적, 산출물 품질 세부 검토, 디버깅

참고: auto 모드에서 maxAssumptions 설정값을 초과하면 자동으로 interactive로 전환됩니다. /u-assume으로 리뷰하면 auto로 복귀합니다.

07. /u-discuss 협업 세션

다자간 토론 세션으로 구조화된 협업을 수행하고, 결과를 SSoT에 자동 반영합니다.

타입용도예시
brainstorm아이디어 발산. 자유로운 제안 수집/u-discuss brainstorm "결제 UX"
review산출물 검토. SSoT 문서 품질 리뷰/u-discuss review
decision기술/비즈니스 의사결정. 선택지 비교 + 결정/u-discuss decision "DB 선택"
workshop다단계 작업. 복잡한 주제를 단계별로 깊이 탐구/u-discuss workshop "메인 화면"
retroIteration 회고. 잘된 점/개선점/액션 아이템/u-discuss retro
Micro-commands (세션 중 사용):
  • @planner / @builder / @gatekeeper / @all — 특정 에이전트에게 질문
  • /idea [text] — 아이디어 태깅
  • /decide [text] — 결정사항 기록
  • /concern [text] — 우려/리스크 기록
  • /action [who] [text] — 액션 아이템 배정
  • /wrap — 세션 종료 + 결과물 SSoT 자동 적재
세션 결과 반영: /wrap으로 종료하면 아이디어는 백로그에, 결정사항은 관련 문서에, 액션 아이템은 태스크로 자동 등록됩니다.

08. 실전 시나리오

Scenario 1: 새 프로젝트 처음부터 끝까지

/u-init my-saas → dropzone에 자료 드롭 → /u-plan --app myapp → /u-design --app myapp → /u-wireframe --app myapp --all → /u-dev --app myapp → /u-check --app myapp → /u-ship --app myapp

한 줄로: /u-loop --app myapp — 전체 파이프라인 자동 실행

Scenario 2: 기존 프로젝트에 SSoT 도입

/u-init → /u-reverse --app myapp → /u-sync myapp → /u-doc myapp srs --edit (비즈니스 의도 보완) → /u-check --app myapp

Scenario 3: 요구사항 변경 대응

/u-add myapp fr "비밀번호 재설정" → /u-update srs --cascade → /u-status myapp (impact flags 확인) → 영향받는 Phase 재실행

Scenario 4: PR 생성

/u-git-pr — 현재 브랜치의 변경사항을 분석하여 커밋 그루핑 + 구조화된 PR body + 리뷰 가이드를 자동 생성합니다.

Scenario 5: Phase Gate 미통과 대응

/u-gate → FAILED (스코어카드 출력) → 미충족 항목 확인 → 해당 문서 수정 → /u-gate (재검증)

예: IA 미완 → /u-plan --app myapp --only ia → /u-gate

Scenario 6: 야간 무인 실행

/u-loop --app myapp — 밤새 돌려놓으면 아침에 결과물을 확인할 수 있습니다.
--from design --to check으로 특정 구간만 지정 가능. 체크포인트 기반 재개 및 에러 자동 복구 지원.

Scenario 7: 멀티 앱 프로젝트

/u-plan common (공통 모듈) → /u-plan web → /u-plan admin → /u-design web,admin -i → /u-sync all (전체 정합성 검증)

09. 폴더 구조

/u-init 실행 시 생성되는 .u-maker/ 디렉터리 구조입니다.

# .u-maker/ 프로젝트 구조
.u-maker/
├── u-maker.config.json         # 전역 설정 (Config v4.0)
├── data/
│   ├── dropzone/                    # Raw 파일 드롭존 (RFP, 회의록, 참고 자료)
│   ├── digest/                      # 분석 결과 (Structured JSON)
│   │   ├── _input/                  # 원시 데이터 파싱 결과
│   │   ├── _classified/             # 12개 카테고리 분류 JSON
│   │   └── _index.json              # 분석 인덱스
│   └── _links.json                  # 글로벌 의존성 그래프
├── docs/{app}/
│   ├── plan/                        # SRS + IA + Roadmap (.md + .json)
│   ├── design/                      # ERD + API + Screens + ScreenFlow + DesignToken + RTM
│   ├── check/                       # Test Cases + Test Results
│   └── _index.json                  # 문서 상태 인덱스
├── output/{app}/

10. Quality Gate (11 Criteria)

Phase 전환 시 Gatekeeper 에이전트가 11개 기준으로 자동 검증합니다. Threshold는 95점이며, 미달 시 개선 항목이 심각도 순으로 표시됩니다.

#기준검증 내용적용 Phase
1SRS 완성도모든 FR/US/FT가 정의되고 상태가 Final인지 확인Plan
2IA 완성도사이트맵/네비게이션 구조가 Screen과 일치하는지 확인Plan
3Roadmap 정합성마일스톤/스프린트 계획이 SRS 범위를 커버하는지 확인Plan
4ERD 완성도엔티티/관계/속성이 SRS 데이터 요구사항을 충족하는지 확인Design
5API Contract 완성도모든 FT에 대응하는 API 엔드포인트가 정의되었는지 확인Design
6Screen 완성도IA의 모든 페이지에 대응하는 Screen spec이 존재하는지 확인Design
7Design System토큰/컴포넌트 정의가 Screen과 일관되는지 확인Design
8RTM 추적성요구사항(FR → US → FT)에서 설계까지 전 구간 추적 가능 여부Design
9Code 완성도Screen/API/ERD 대비 코드 커버리지. spec-sync 검증 통과 여부Dev
10TC 커버리지모든 FT에 대응하는 TestCase가 존재하고 실행되었는지 확인Check
11결함 ZeroCritical/Major 결함이 0인지 확인. Minor는 백로그 등록 후 통과 가능Check
Gate 명령어: /u-gate를 실행하면 현재 Phase의 해당 기준을 검증하고, 스코어카드를 출력합니다. PASS(>=95점) 시 다음 Phase로 자동 전환됩니다. FAIL 시 미충족 항목과 개선 방법이 심각도 순으로 표시됩니다.

11. FAQ

  • Q
    명령어가 인식되지 않아요
    Claude Code를 재시작해보세요. 그래도 안 되면 claude plugin add upleat-ax/u-maker-plugin으로 재설치하세요.
  • Q
    어떤 명령어를 써야 할지 모르겠어요
    자연어로 말하세요. PM Agent가 자동으로 적절한 커맨드로 라우팅합니다. 또는 /u-ask로 질문하세요.
  • Q
    기존 프로젝트에 적용하려면?
    /u-init → /u-reverse로 코드에서 SSoT를 역공학하세요. --loop 옵션으로 Gate 검증까지 포함 가능합니다.
  • Q
    문서 언어를 바꾸고 싶어요
    u-maker.config.json의 language.documents를 "en", "ja", "zh" 등으로 변경하세요.
  • Q
    auto 모드에서 자꾸 interactive로 전환돼요
    maxAssumptions 설정값 초과 시 발생합니다. /u-assume으로 미결 가정을 approve/reject 하세요.
  • Q
    Gate가 계속 FAIL이에요
    스코어카드의 미충족 항목을 확인하세요. Threshold는 95점입니다. 해당 문서를 수정하거나 --only 옵션으로 특정 산출물만 재생성한 후 /u-gate를 재실행하세요.
  • Q
    /u-loop 실행 중 에러가 나면?
    체크포인트 기반 재개를 지원합니다. 동일한 /u-loop 명령을 다시 실행하면 마지막 체크포인트부터 이어서 진행합니다.
  • Q
    PR을 빠르게 만들고 싶어요
    /u-git-pr — 현재 브랜치의 변경사항을 분석하여 커밋 그루핑 + 구조화된 PR을 자동 생성합니다.
  • Q
    이전 버전(v3.x)에서 마이그레이션하려면?
    /u-init --migrate를 실행하세요. 기존 .u-maker/ 구조를 Config v4.0으로 업그레이드합니다.
